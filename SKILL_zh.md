# Temporal Search Enhancer — 时间感知搜索增强

## 核心理念

大模型缺乏内在时间感——它不知道"现在"是什么时候，也不理解"最近"对用户意味着哪个具体时间段。这导致搜索query缺少时间约束，搜索引擎返回过时或错位的结果。

本skill是一个**纯推理中间层**，在用户原始query和搜索执行之间插入时间推理，让每一条搜索query都带上正确的时间锚点。

---

## 执行流程

收到用户查询后，按以下四步顺序执行。每一步的输出喂入下一步。

### Step 1: 时间锚定（Time Anchoring）

将"现在"具象化。基于系统提供的当前日期，建立以下上下文变量：

```
current_date: 2026-04-25        # 当前精确日期
current_year: 2026
current_quarter: Q2
current_month: April
current_weekday: Saturday
```

同时推导出与查询主题相关的**周期锚点**，例如：
- 体育赛事：当前处于哪个赛季？（如"2025-2026赛季"、"2026春季赛"）
- 展会会议：最近一届是哪年？下一届何时？
- 周期活动：上一个周期实例何时？下一个何时？

周期锚点不需要精确到日——年份或赛季级别即可，目的是帮助后续步骤正确定位。

### Step 2: 时间意图识别（Temporal Intent Classification）

判断用户query的时间指向，分为以下类别：

| 意图类型 | 含义 | 典型信号词 | 示例 |
|---------|------|-----------|------|
| **realtime** | 需要此刻/当天数据 | "比分""直播""现在" | "LOL比赛比分" |
| **recent** | 需要近期（可回溯）数据 | "最近""最新""近期" | "最近的AI新闻" |
| **upcoming** | 需要未来事件信息 | "下一届""什么时候""即将" | "下一届奥运会在哪" |
| **periodic_current** | 定位到当前/最近周期实例 | 周期性事件名本身 | "双十一优惠""春季赛" |
| **historical** | 明确指向过去某个时间点 | "去年""2020年""上一届" | "2022世界杯冠军" |
| **timeless** | 不需要时间约束 | 无时间信号 | "什么是量子计算" |

**判断规则：**
- 如果query中有显式时间词，直接使用对应类型
- 如果没有显式时间词，但查询对象本身有强时间属性（赛事、新闻、政策等），默认为 `periodic_current` 或 `recent`
- 只有查询对象确实与时间无关时（纯概念、纯定义），才归为 `timeless`
- 拿不准时，宁可加时间约束也不要漏掉——多一个年份关键词的代价远小于返回过时结果

### Step 3: 查询时间增强（Query Time Augmentation）

根据Step 1的锚点和Step 2的意图，对用户原始query进行改写或补充。

**增强策略：**

| 意图类型 | 增强方式 | 示例（假设当前2026年4月） |
|---------|---------|------------------------|
| realtime | 加"今天"/"实时"/"live" + 精确日期 | "LOL比赛" → "LOL 英雄联盟 比赛 2026年4月25日 赛果" |
| recent | 加近期时间范围 | "AI新闻" → "AI新闻 2026年4月" |
| upcoming | 加"即将"/"upcoming" + 年份 | "奥运会" → "2028奥运会 赛程 举办地" |
| periodic_current | 加当前周期标识 | "英雄联盟比赛" → "英雄联盟 2026 春季赛 / MSI 2026" |
| historical | 保留用户指定的时间 | "2022世界杯冠军" → 原样保留 |
| timeless | 不加时间约束 | "量子计算原理" → 原样保留 |

**增强原则：**
- 生成 2-3 条互补的增强query，覆盖不同角度（如赛果、赛程、排名）
- 时间标识要用搜索引擎友好的格式：年份用数字（2026），月份可用数字或文字
- 保留用户原始query中的核心关键词，不要过度改写语义
- 对于周期性赛事，同时使用通用名称和官方赛事名（如"英雄联盟"+"LPL"+"MSI"）

### Step 4: 结果新鲜度排序建议（Freshness Priority）

为搜索结果的后处理提供时间维度的排序建议：

| 意图类型 | 新鲜度策略 |
|---------|-----------|
| realtime | 优先最近24小时内容，次优先最近一周，不丢弃更早的相关背景 |
| recent | 优先最近一个月内容，次优先最近三个月 |
| upcoming | 优先包含未来日期的内容 |
| periodic_current | 优先当前周期（如当前赛季）内容 |
| historical | 优先与指定时间匹配的内容 |
| timeless | 无时间偏好，按相关性排序 |

注意：这里是**优先级排序**而非硬性过滤。即使突发新闻场景，也不要丢弃24小时之外的相关背景内容，只是降低其排序权重。

## 输出格式

完成四步分析后，输出以下结构化结果（供下游搜索Agent或系统使用）：

```json
{
  "original_query": "用户原始查询",
  "current_date": "2026-04-25",
  "temporal_intent": "periodic_current",
  "time_anchor": {
    "description": "2026赛季春季赛/MSI阶段",
    "primary_period": "2026",
    "sub_period": "Spring Split / MSI"
  },
  "augmented_queries": [
    "英雄联盟 LPL 2026 春季赛 战绩 排名",
    "LOL MSI 2026 赛程 赛果",
    "英雄联盟 2026 最新比赛结果"
  ],
  "freshness_priority": {
    "level": "recent",
    "preferred_window": "最近3个月",
    "fallback_window": "2026年全年"
  },
  "reasoning": "用户查询'英雄联盟赛况'未指定具体时间，但赛事属于强时间属性主题..."
}
```

## 特殊场景处理

### 跨年/跨赛季边界
赛事赛季往往跨年（如2025-2026赛季）。当前日期处于赛季中间时，同时覆盖赛季起止时间。

### 多赛事体系
像英雄联盟这样有多级赛事（LPL/LCK/LEC联赛 → MSI → 全球总决赛）的项目，根据当前日期判断正在进行或刚结束的赛事级别，优先查询该级别。

### 地域差异
同一时间词在不同地域可能指不同事件。如"联赛"在中国语境更可能指LPL，在韩国更可能指LCK。结合 user_locale 判断。

### 模糊意图
当无法确定用户意图是回顾还是前瞻时（如"世界杯"在非赛事年），同时生成回顾上届和展望下届的query，让搜索结果自然覆盖。

---

## 与搜索工具的配合

本skill的输出可以直接指导以下搜索行为：
1. 将 `augmented_queries` 中的query依次传入搜索工具
2. 对返回结果按 `freshness_priority` 进行排序
3. 在回答用户时，标注信息的时间点，帮助用户判断时效性

如果搜索工具支持时间范围过滤参数（如Google的 `tbs=qdr:m`），优先使用；否则通过query关键词隐式约束。
